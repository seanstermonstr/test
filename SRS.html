1.0 Introduction

1.1 The purpose of this document

This is the Drunken Adventure software requirement specification document for the Eagle Day Software. This document will provide a general description of the game along with the software requirements. The final product will be traceable to this requirements document.

1.2 The scope of this product

The Eagle Day Software is a 32-bit Windows application. The application targets users who are familiar with the rules. The application replicates the historical simulation board game Eagle Day. The application is written in Lua using the Love library.

2.0 General Description

2.1 Procuct Perspective

Eagle Day is a multimedia application that replicates the historical simulation board game, “Eagle Day: The Battle For Britain” designed by Decision Games. Eagle Day provides a means of playing Eagle Day: The Battle For Britain. It allows two players to play against each other via network or locally on the same computer. Eagle Day also allows a single player to play against a computer.

2.3 User Characteristics

Users are expected to be able to user a mouse and navigate through a Windows environment in order to use Eagle Day. Users are expected to either already know and understand the rules of Eagle Day: The Battle of Britain or learn for them self using the rules that will be provided in the application.

2.4 General Constraints

Eagle Day shall be designed to run on a computer running a version of Windows 7 or Window 8. The computer should be compatible with the LÖVE framework.

2.5 Assumptions and Dependencies

The Eagle Day application assumes that, or depends upon, the following environmental characteristic being present for full functionality:

  1. The General Constraints listed in section 2.4 of the SRS are met
  2. The customer shall have permission to install the Eagle Day application on the required workstation(s).

3.0 Software Requirements

3.1 Behavioral Requirements

This section of the SRS provides the behavioral requirements of Eagle Day

3.1.1 Graphical User Interface

  • 3.1.1.1: The display should respond to mouse movements.
  • 3.1.1.2: The GUI will be coded using LOVE.
  • 3.1.1.3: When the Eagle Day executable file is run, there will be a graphical user interface.
  • 3.1.1.4: The GUI will involve a realistic map of the board.
  • 3.1.1.5: There will be players that will be easy to differentiate.

4.0 System Features

4.1 New Game

The title screen will give the suer the option to start a new game and specify how and who they will play against.

4.1.1 Response Sequences

UC 01 The user starts a new game.
Description The application will initiate a new game.
Precondition The application is running and is on the title screen or in-game menu screen.
Post-condition The application will prompt the user to choose how and who they will play against.
Normal Sequence
  • The user starts the application.
  • The user clicks New Game.
  • The application initiates a new game from the title screen or the in-game menu screen.
  • The application prompts the user to save the current game (if in game).
  • The application prompts the user to select how and who they will play against.

UC 02 The user selects Network Play for two human players.
Description The application initiates a new game against another human over a network.
Precondition The computer has network connection capabilities and the user selected the option to start a new game.
Post-condition An opponent is found on a network and the application initiates a new game ready to be played.
Normal Sequence
  • The user selects New Game.
  • The user selects to play over a network with another human player.
  • The application searches for an opponent.
  • The application finds an opponent.
  • The application initiates a new game ready to start.

UC 03 The user selects Face-To-Face play for two human players.
Description The application initiates a new game for two humans on the same computer.
Precondition The user selected New Game.
Post-condition The application initiates a new game to be played by two humans on the same computer.
Normal Sequence
  • The user selects New Game.
  • The user selects Face-To-Face.
  • The application initiates a new game ready to start.

UC 04 The user selects Single Player to play against an AI.
Description The application initiates a new game for one player to play against an AI computer.
Precondition The user selects New Game.
Post-condition The application initiatesa new game ready to be played.
Normal Sequence
  • The user selects New Game.
  • The user selects Single Player.
  • The application initiates a new game ready to start.

4.1.2 Functional Requirements

  • REQ-1: The application shall be able to run.
  • REQ-2: The application shall be able to initiate a new game.
  • REQ-3: The application shall be able to connect to a network and find another player.

4.2 Save Game

In the in-game menu, the user shall have the option to save their current game.

4.2.1 Response Sequences

UC 05 The user saves the game.
Description The application will ask to save the current game if necessary from a dialog, a save game file is picked and loaded.
Precondition A game is in progress.
Post-condition The current game state is saved into memory.
Normal Sequence
  • The user is playing the game.
  • The user opens the in-game menu and selects Save Game.
  • The application opens a dialog to save the game with the option to pick the name of the file.
  • The application returns to the previous screen (Title or in-game menu).

4.2.2 Functional Requirements

  • REQ-4: The application shall be able to save the current game information to a file.

4.3 Load Game

The user shall be able to load a previously saved game from a file.

4.3.1 Response Sequences.

UC 06 The user loads a saved game file.
Description The application will ask to save the current game if necessary from a dialog, a saved game file is picked and loaded.
Precondition A saved game file is available
Post-condition The application has loaded the saved game file and is being played.
Normal Sequence
  • The user is on the title screen or in-game menu.
  • The user selects Load Game.
  • The application opens a dialog asking to save the current game if applicable and saves if selected.
  • The application opens a dialog to load a saved game file from the computer.
  • After a saved file is picked the appilcation loads the file.
Exception
  • Condition: No save file is selected
  • Action: The application waits fro teh user to cancel out of the dialog box.
  • Ending: The application returns to the pause menu (if in game) or main menu (not in game).

4.3.2 Functional Requirements

  • REQ-5: The application shall be able to load information from a saved game file.

4.4 Interactive Game Board

The main feature in the game will be the game board. The game board will feature similar pieces, campaign cards, and map to the Eagle Day: Battle For Britain board game.

4.4.1 Response Sequences.

UC 07 The user selects and moves a piece.
Description The application changes the position of the piece to the newly selected location.
Precondition A game is in progress.
Post-condition The application will show the new position of the moved piece.
Normal Sequence
  • The user selects and moves a piece.
  • It is the users turn and the user selects a piece and where to move it.
  • The application changes the position of the piece.

UC 08 The user uses a unit to attack another unit.
Description The application calculates the outcome of attack.
Precondition A game is in progress.
Post-condition The outcome of attack is displayed.
Normal Sequence
  • The user is playing the game.
  • It is the user's turn and a valid attacking phase.
  • The user selects a unit to attack with.
  • The user selects a valid unit to attack.
  • The application calculates and displays the outcome of attack.

UC 09 The user plays a campaign card.
Description The application performs the specifics of the campaign card.
Precondition A game is in progress. The player has a campaign card to play.
Post-condition The results of the campaign card are applied and displayed.
Normal Sequence
  • The user is playing the game.
  • It is the user's turn and has a campaign card to play.
  • The user selects a campaign card to play
  • The application appies and displays the specifics of the campaign card.

4.4.2 Functional Requirements

  • REQ-6: The application shall be able to display and change the location of the game pieces.
  • REQ-7: The application shall be able to store in-game stats, change them, and display changes.

4.5 Rules

The user shall be able to access the game rules from the title screen or the in-game menu.

4.5.1 Response Sequences

UC 10 The user accesses the game rules.
Description The application will display the game rules.
Precondition The application is running and is on the title screen or in-game menu.
Post-condition The game rules document is displayed.
Normal Sequence
  • The application is running
  • The user accesses the game rules.
  • The application displays the game rules document.

4.5.2 Functional Requirements

  • REQ-8: The application shall contain an accessable game rules document.

4.6 Exit Game

The user shall be able to exit the application from in-game or the Title page.

4.6.1 Response Sequences

UC 11 The player wants to quit the application.
Description The application quits running.
Precondition The application is running.
Post-condition The application is closed.
Normal Sequence
  • The application is running.
  • The player selects exit from either in-game or the Title Page.
  • The application prompts the user to save the current game if applicable.
  • The application closes.

4.6.2 Functional Requirements

  • REQ-9: The game shall be able to close.
